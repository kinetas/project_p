# 2. 네이밍 규칙

## 변수명
- camelCase 사용 (Java 표준)
- 예: `stockPrice`, `debtRatio`, `currentUser`

## 함수(메서드)명
- camelCase 사용 (Java 표준)
- 예: `getStockList()`, `calculatePER()`, `findByStockCode()`

## 클래스명
- PascalCase 사용 (Java 표준)
- 예: `StockController`, `FinancialDataService`, `StockEntity`

## 상수명
- UPPER_SNAKE_CASE 사용
- 예: `MAX_RETRY_COUNT`, `DART_BASE_URL`

## Entity 필드명
- Java 필드는 camelCase 사용
- DB 컬럼명은 snake_case로 명시 (`@Column(name="snake_case")` 어노테이션 필수)
- 예:
  ```java
  @Column(name = "stock_code")
  private String stockCode;

  @Column(name = "debt_ratio")
  private Double debtRatio;
  ```

# 3. 패키지 & 폴더 구조

```
develope/backend/src/main/java/com/example/demo/
├── controller/    # HTTP 요청 처리, REST 엔드포인트
├── dto/           # 요청/응답 데이터 전송 객체
├── service/       # 비즈니스 로직
├── entity/        # JPA 엔티티 (DB 테이블 매핑)
├── repository/    # DB 접근 (JPA Repository)
└── config/        # 설정 클래스 (CORS, Scheduler 등)
```

---

# 4. 레이어 역할 분리

- Controller: 요청 수신 및 응답 반환만 담당. 비즈니스 로직 작성 금지
- Service: 핵심 비즈니스 로직 및 지표 계산 처리
- Repository: DB 쿼리만 담당. 로직 작성 금지
- DTO: Entity를 직접 반환하지 않고 DTO로 변환하여 반환
- Entity: DB 테이블과 1:1 매핑. 비즈니스 로직 포함 금지

---

# 5. API 설계 규칙

- REST 방식 사용
- URL은 소문자 + 하이픈(kebab-case) 사용
  - 예: `/api/stocks`, `/api/stocks/{code}/financials`
- HTTP 메서드 의미에 맞게 사용 (GET 조회, POST 생성, PUT 수정, DELETE 삭제)
- 응답은 JSON 형식

---

# 6. 외부 API 키 관리

- API Key는 `src/main/resources/application.properties`에서 관리
- Git에는 키 값을 비운 양식만 커밋
  ```properties
  dart.api.key=
  krx.api.key=
  ```
- 코드 내 하드코딩 금지. `@Value` 어노테이션으로 주입
